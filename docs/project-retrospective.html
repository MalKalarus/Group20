PROJECT      RETROSPECTIVE

Workflow:

We attempted to use an AGILE process for software development.  By continually meeting with project stakeholders and team members on a weekly, if not daily, basis, we were able to update and change our project requirements as new developments (such as coding challenges, time limitations, or a new understanding of stakeholder expectations) shaped our project.   Early on, we produced a prototype of our project and continuously increased the complexity and dependability of the software being delivered to our clients.  Team members continually reflected amongst themselves on how to become more effective, particularly in regard to Agile’s tenet of simplicity.

Team 20 used a public repository on Github to share our coding developments with each other and other project stakeholders like Ethamajin Ltd..  The integration of Github with our IDE, IntelliJ, allowed for easy pushing and pulling of new commits.  The master branch of our github repository was reserved for software releases.  Tasks were envisioned as github branches and assigned to team members, who could then fork off their own branch for minor adjustments to their code.  Pushing code was done after consultation with other team members -- ideally face-to-face, so integration issues could be discussed in real-time.  Task distribution was accomplished through assignment of tasks at meetings, and through Trello.  Assigning relevant tasks was difficult due to conflicting perspectives on the final design of the project and uncertainty regarding steps towards project progression.  

Development tools and frameworks used by Team 20 include Grails, React, and the fetch API.

MVC Design Pattern:

​Team 20 followed a model-view-controller design (MVC) in designing ResuMe.  Data was stored in a GORM database in Grails and named by our domain classes. Methods to manipulate data and communicate between front and back end were placed in our controller classes.  Our views were given through the interface, produced via .html, .css, and React files coupled with the .gsp files native to Grails.  As an example of how our classes interact, if a user tried to sign up through our interface, our UserAccount controller would fetch the username and password from the front end, and then check the former against preexisting usernames in our database.  If the username did not exist in our database, the UserAccount controller would then create a new UserAccount object with the given information, and store it in our database.  Our server-side program was implemented as a REST api using Grails.
Refactor Retrospective:

The Good:
The organization of our original UML was excellent, with dependencies between classes clearly outlined.  Example:  The UserAccount class had a Resume, which had dependent components such as address, application, qualification, et cetera..

The Bad:
During our development process, we found that our program crashed very often. A domain class generated to pull the exhibit the a user’s data for presentation purposes, ActiveUser, worked occasionally and failed at other times for unknown reasons. The performance of SpringSecurity plugin was inconsistent in spite of proper installation. These unsolved and rather unreasonable happenings troubled us with running the program smoothly.
Project Retrospective:

The design of what GUI was completed was impeccable.  We could have had a better correspondence between fields on our front-end forms and domain objects on the back-end, however.  The largest problem with our project was lack of completion, partly due to lack of familiarity with software developments and our chosen framework, partly due to lack of communication.  If we were to relaunch ResuMe, we would choose a simpler framework to work with, such as Python/Flask.  Our team members would spend more time reading up on basic concepts like HTTP methods that are integral to how web applications work, since most of us were unfamiliar with web development before this course. We would also ensure to commit our code to our Github repository very early in each development stage, to allow for more help from our peers, and to resolve integration issues before submitting our project for evaluation.
